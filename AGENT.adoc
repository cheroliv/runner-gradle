== Projet

**Nom** : runner-gradle — point d'entree unique du workspace, build script only
**Stack** : Kotlin DSL Gradle, `includeBuild()` pour plugins N0/N1/N2 en dev mode
**Role dans le DAG** : N3 — Orchestration (voir `AGENT_GOVERNANCE.adoc` → Regle 4bis)
**N'EST PAS** un plugin Gradle — pas de `src/main/kotlin`, pas de `publishToMavenLocal`

Le projet `runner-gradle/` orchestre la flotte de plugins Gradle (N0→N2) depuis un seul
`build.gradle.kts`. Il sert d'interface CLI pour le developpeur et d'API pour le SaaS Edster.

Consomme les plugins de `foundry/` comme dependances ou `includeBuild`, lit les secrets
depuis `configuration/` (cercle 1), et applique les regles de gouvernance globale.

== Regles Absolues

=== Git

**INTERDICTION FORMELLE** de commit/push/merge sans permission explicite de l'utilisateur.

Commandes autorisees (lecture seule) : `git status`, `git diff`, `git log`, `git show`
Commandes interdites (sauf ordre) : `git add`, `git commit`, `git push`, `git merge`, `git rebase`

=== Pas de `publishToMavenLocal`

Runner n'est pas un plugin. Il n'y a rien a publier. La seule commande Gradle legitime
est l'execution de taches (`./gradlew aggregateGraphs`, `./gradlew verifyDagAcyclic`, etc.).

=== Securite

* Secrets lus depuis `configuration/` (cercle 1), jamais stockes ici
* Zéro credential dans `build.gradle.kts`

=== Tests

L'agent NE DOIT JAMAIS lancer de tests lors de la procedure de fin de session SANS permission explicite.

*REGLE FONDAMENTALE — Runner-gradle n'a pas de tests.*

Runner-gradle est un *terminal consommateur*. Il n'implémente aucune logique métier.
Toute logique (et donc tout test) appartient au plugin propriétaire :
- `verifyDagAcyclic` → testé dans graphify-gradle (N0)
- `augmentOpencode` → testé dans codebase-gradle (N1)
- `collectCompositeContext` → testé dans codex-gradle (N2)

Runner n'a ni `src/test/`, ni framework de test, ni besoin de tests.
La charge de la preuve est sur le plugin, pas sur le consommateur.

=== REGLE 6 — Ne pas ajouter de tâches hors contrat sans validation explicite

Les 4 tâches cibles sont définies dans la section <<Taches Cibles>>. Toute nouvelle
tâche (serveur HTTP, endpoint REST, etc.) doit être proposée et acceptée avant
implémentation. Voir leçon L09 dans `configuration/.agents/lazy/lecon-l09.adoc`.

=== Regle 7 — Verifier la compilation AVANT chaque commit

* Avant `git commit`, lancer la tache concernee (ex: `./gradlew <nomDeLaTache>`)
* Si echec -> interdiction de commiter -> corriger d'abord
* Voir lecon L10 dans `configuration/.agents/lazy/lecon-l10.adoc`

== Procedure de Fin de Session (obligatoire)

. Executer les 6 etapes standard pour `foundry/runner-gradle/`

== Contrat DAG — N3 Orchestration

Runner est le seul projet N3. Regles absolues du DAG (voir `AGENT_GOVERNANCE.adoc` Regle 4bis) :

* *Peut importer* : tous les N0 (`graphify-gradle`), N1 (`codebase-gradle`), N2 (`plantuml-gradle`, `slider-gradle`, `bakery-gradle`, etc.)
* *Ne peut JAMAIS etre importe* : par N0, N1, ou N2. Aucun plugin du workspace ne depend de runner-gradle.
* *Est le consommateur du graph.json global* : runner-gradle agrege tous les `graph.json` locaux en un graphe unifie

== Taches Cibles

Runner N3 ne contient plus de logique metier — toutes les verifications, aggregations
et transformations sont deleguees aux plugins propriétaires (N0/N1/N2).

[cols="1,3,3"]
|===
| Tache | Description | Fournie par
| `scanWorkspace` | Scan workspace → graph.json (node/edge/community) | `graphify-gradle` (N0)
| `verifyDagAcyclic` | Verifie qu'aucun plugin n'importe un plugin de niveau superieur | `graphify-gradle` (N0)
| `deployWorkspace` | Provisionne un workspace client complet (tous les plugins + office/ + configuration/) | runner-gradle (N3)
| `verifyBoroughs` | Valide l'arborescence, la gouvernance agent et les metadonnees des 11 boroughs (EPIC K-4 Convention over Configuration) | runner-gradle (N3)
|===

Le contrat DAG : la charge de la preuve est sur le plugin proprietaire, pas sur le consommateur.
